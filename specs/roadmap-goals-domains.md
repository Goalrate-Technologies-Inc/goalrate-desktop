# Roadmap, Goals, Domains, Tasks

## Problem

The Roadmap needs to turn a user's goals into an organized, actionable structure. Users should be able to see goals grouped by life/work domain, understand priority, and drill down into tasks and subtasks that drive daily planning.

## Goals

- Define the product vocabulary for Domain, Goal, Task, and Subtask.
- Make Roadmap display goals grouped by Domain.
- Show color-coded Goal priority indicators using Low, Medium, High, and Critical.
- Persist user-facing entities in markdown-readable form.
- Define what happens when tasks or subtasks repeatedly fail to get completed.
- Clarify that product specs use `Goal` and `Domain` even if code currently uses internal/legacy names.

## Non-Goals

- Do not define public Web goal publishing in this spec.
- Do not define project management features unrelated to goals, tasks, and subtasks.
- Do not require full task dependency graphs for v1.
- Do not automatically delete user goals because the AI classified them as low value.

## User Experience

The Roadmap is a panel or view that lists Domains. Each Domain can be expanded or collapsed and contains its Goals. Each Goal shows enough metadata to scan:

- Title.
- Status.
- Deadline, if any.
- Color-coded Goal priority.
- Progress/success metric, if available.
- A quick path to tasks, notes, and Assistant actions.

Users can create, edit, archive, or delete goals. The Assistant can add new Goals and edit existing Goal metadata from chat when the user asks it to update the Roadmap. GoalRate and the Assistant can generate tasks for a goal. When a task repeatedly fails, GoalRate should break it down into subtasks or ask the user whether the work is still worth pursuing.

## Requirements

### Domain

A Domain is a grouping for related Goals, such as Startup, Health, Family, Career, or Personal.

Domain behavior:

- Domains have a name and optional icon.
- Domains can be toggled open or closed in the Roadmap.
- In v1, Domains are inferred from Goal frontmatter rather than requiring separate Domain files.
- Optional/deferred Domain metadata files may define sort order, color, icon, archive state, and urgency/importance thresholds.
- Domains can be archived without deleting contained goals.
- Goals can be moved between domains.

### Goal

A Goal is an objective specified by the user during onboarding or by chatting with the Assistant.

Goal behavior:

- Goals belong to one Domain.
- Goals have lifecycle state.
- Goals can contain or reference Tasks.
- Goals may have a deadline and success metric.
- Goals have one local owner in v1.
- Goal frontmatter may include an empty `collaborators` field for forward compatibility, but local collaborator workflows are deferred.
- Goals may be visible locally only in v1. Publishing is future/deferred.
- Assistant-created Goals must be persisted as normal Goal markdown files in `goals/` with `Actor: assistant` mutation-log attribution.
- Assistant edits to Goals must be limited to explicit user-requested metadata changes such as title, Domain, deadline, success metric, priority, status, and notes.
- Assistant Goal edits must preserve existing embedded Tasks/Subtasks unless the user separately asks for a Task/Subtask change.
- The Assistant must not delete a Goal, archive a Goal, abandon a Goal, or otherwise perform destructive Roadmap cleanup without explicit user confirmation.

Goal lifecycle:

- `created`: captured but not yet actively pursued.
- `active`: available for Roadmap and Agenda planning.
- `paused`: retained but excluded from normal daily planning.
- `completed`: successfully finished.
- `abandoned`: user decided not to pursue it.
- `archived`: hidden from active Roadmap views, retained for history.

Failure handling for goals should follow this coaching order:

```text
reschedule -> break down -> ask -> coach -> archive
```

The app should not move a goal to `abandoned` or `archived` without user confirmation.

### Task

A Task is an actionable item generated by the AI or created by the user for a Goal. Tasks can be added to the Agenda.

Task behavior:

- Tasks have a parent Goal.
- Tasks remain embedded in their parent Goal markdown file.
- Tasks have status and optional due date.
- Tasks may have effort estimate, energy level, blocker, recurrence, and an exact scheduled date.
- In the Roadmap Goal detail task list, users can make a Task recurring, choose the recurrence frequency, and set or clear the exact date the Task belongs on an Agenda.
- Tasks can be completed, deferred, edited, archived, or deleted.
- Completed non-recurring tasks should not be planned again.
- Completed recurring tasks may appear again according to their recurrence rule.
- Complex Tasks should be broken down into Subtasks rather than moved into separate markdown files.
- Desktop MVP task candidate collection must use Goal-embedded Tasks/Subtasks and must not load standalone legacy `projects/` task files into Agenda, Focus, Roadmap, or Assistant planning context.
- If a task is on the Agenda and not completed within 2 days, the AI should break it into Subtasks unless it already has subtasks.

### Subtask

A Subtask is a smaller action generated from a Task or created by the user.

Subtask behavior:

- Subtasks belong to a parent Task.
- Subtasks should describe a concrete first physical action where possible.
- If a subtask is on the Agenda and not completed within 2 days, the app should ask whether the user wants to continue the Subtask, Task, or Goal.

Subtask 2-day failure flow:

```text
Subtask not completed within 2 days on Agenda
-> Ask: Continue with subtask?
   -> Yes: add subtask to tomorrow's Agenda.
   -> No: ask: Continue with parent task?
      -> Yes: ask: Generate a different subtask?
         -> Yes: AI generates a new subtask for the task.
         -> No: add parent task to tomorrow's Agenda.
      -> No: ask: Generate a different task for the goal?
         -> Yes: AI generates a new task for the goal.
         -> No: ask: Continue with goal?
            -> Yes: remove or archive the task from the goal.
            -> No: archive or abandon the goal after explicit confirmation.
```

For the desktop MVP, replacement Subtask and replacement Task generation may use a deterministic local first-step generator when a remote AI call is unavailable or unnecessary. The result must still be treated as an Assistant-originated mutation when surfaced through the Assistant flow: validate frontmatter, write to the vault, preserve the missed Subtask and existing parent Task, and log the mutation.

When the user chooses a different Subtask, parent Task, different Task, Goal continuation, or Goal archive from this flow, GoalRate should record `last_missed_decision_on` on the missed Subtask. This marker resolves the current prompt without pretending the missed Subtask was completed or deleting it.

For the desktop MVP, continuing with the Goal means preserving the active Goal and archiving the missed Subtask's parent Task branch after explicit user confirmation. This is a non-destructive frontmatter update: the Goal file remains in place, the Task/Subtask records remain in markdown with `status: archived`, and the mutation is logged as Assistant-originated.

For the desktop MVP, not continuing with the Goal means archiving the Goal after explicit user confirmation. This is also non-destructive: the Goal file remains in place with `status: archived` and `lifecycle: archived`, embedded Tasks/Subtasks remain in markdown, and the mutation is logged as Assistant-originated.

### Roadmap Display

- Group Goals by Domain.
- Domains are sorted by optional metadata `sort_order` when present, then name.
- Archived Domains and Goals are hidden by default but reachable from filters/settings.
- Goal priority indicators should reflect the Goal's `priority` field.
- Goal priority indicators should use Low, Medium, High, and Critical labels.
- Goal priority indicator colors should be Critical red, High orange, Medium yellow, and Low blue.
- Color must not be the only signal. Compact indicators should expose the priority label through a `title` attribute and accessible text.
- The Roadmap should refresh when Goal or Task-relevant vault markdown changes.
- The Roadmap should use vault watcher path payloads when available so unrelated changes, such as `logs/errors.md` or Agenda-only edits, do not force a Roadmap reload.
- In the Goal detail modal, users should be able to adjust the Goal priority without closing the modal.
- In the Goal detail modal, clicking the notes section, including the empty notes state, should make notes editable inline while keeping the user in the Goal detail view. The inline editor should be a focused textarea with Save/Cancel controls, not a separate edit view with Edit/Preview tabs or a repeated Goal Notes header. After Save, the notes section should return to the normal rendered notes view.

## Public Interfaces / Data Model

### Product Naming

Product-facing specs use:

- `Domain`: user-facing grouping.
- `Goal`: user-facing objective.
- `Task`: user-facing action for a Goal.
- `Subtask`: smaller action under a Task.

Current implementation may use:

- `Objective` as an internal alias for `Goal`.
- `goal_type` or `type` as an internal alias for `Domain`.

Implementation should avoid exposing legacy/internal names in new user-facing copy.

### Optional/Deferred Domain Frontmatter

Suggested file: `domains/<domain-slug>.md`

GoalRate should not require these files for v1. When they are absent, the Roadmap groups Goals by `domain`, by the current internal `type`/`goal_type` field, or by a legacy tag fallback.

```yaml
---
id: domain_startup
type: domain
name: Startup
icon: rocket
color: "#2563eb"
sort_order: 10
urgency_threshold:
  due_within_days: 7
  overdue_is_urgent: true
importance_threshold:
  min_goal_priority: high
  requires_success_metric: false
archived: false
created: "2026-04-25T12:00:00Z"
updated: "2026-04-25T12:00:00Z"
---
```

Required fields when a Domain metadata file exists:

- `id`
- `type: domain`
- `name`

Optional fields:

- `icon`
- `color`
- `sort_order`
- `urgency_threshold`
- `importance_threshold`
- `archived`
- `created`
- `updated`

### Goal Frontmatter

Suggested file: `goals/<goal-slug-or-id>.md`

```yaml
---
id: goal_launch_mvp
type: goal
title: Launch GoalRate MVP
domain_id: domain_startup
domain: Startup
status: active
visibility: private
deadline: "2026-06-01"
success_metric: "MVP released to first private beta users"
owner: local_user
collaborators: []
priority: high
tags:
  - startup
lifecycle: active
created: "2026-04-25T12:00:00Z"
updated: "2026-04-25T12:00:00Z"
tasks:
  - id: task_finish_onboarding
    title: Finish onboarding flow
    status: todo
    effort_estimate_minutes: 90
    due_date: "2026-04-27"
    energy_level: high
    blocker: null
    recurrence: none
    parent_goal_id: goal_launch_mvp
    eisenhower_quadrant: do
    subtasks: []
---
```

Required fields:

- `id`
- `type: goal`
- `title`
- `status`

Recommended fields:

- `domain_id` or `domain`
- `visibility`
- `deadline`
- `success_metric`
- `owner`: one local owner in v1.
- `collaborators`: optional forward-compatible field; should remain empty until collaborator workflows exist.
- `priority`
- `lifecycle`
- `tasks`
- `created`
- `updated`

### Task Fields

Tasks must remain embedded in their parent Goal file. GoalRate should not split complex Tasks into separate task markdown files; it should represent added complexity with Subtasks under the embedded Task.

GoalRate must validate embedded Task and Subtask rows before typed loading. A malformed row should be reported to `logs/errors.md` with the affected Goal file and row-level field path, and the raw Goal markdown must be preserved without read-time rewrites.

Required fields:

- `id`
- `title`
- `status`
- `parent_goal_id`

Recommended fields:

- `effort_estimate_minutes`
- `due_date`
- `energy_level`
- `blocker`
- `recurrence`
- `eisenhower_quadrant`
- `first_seen_on_agenda`
- `last_seen_on_agenda`
- `last_missed_decision_on`
- `deferral_count`
- `completed_at`
- `subtasks`

Allowed `status` values:

- `todo`
- `in_progress`
- `deferred`
- `blocked`
- `completed`
- `archived`

Legacy compatibility aliases may be read from older local vault data:

- `pending`: treat as active/pending work equivalent to `todo`.
- `done`: treat as completed work equivalent to `completed`.
- `cancelled`: treat as inactive work equivalent to `archived`.

New GoalRate writes should use the canonical values above.

Roadmap-supported recurrence frequency values:

- `none`
- `daily`
- `weekdays`
- `weekly`
- `monthly`
- `yearly`

For compatibility with existing vault data, the desktop app reads both `recurrence` and the legacy/current `recurring` task field. The Roadmap selector currently writes the selected frequency to `recurring` until the embedded Task schema is migrated to one canonical key.

When GoalRate writes a new child Task/Subtask row that points at a parent Task, it should include `generated_from_task_id` with the parent Task id. It may also keep `parent_id` for compatibility with older flat child rows.

### Subtask Fields

Subtasks may be embedded under a Task.

Required fields:

- `id`
- `title`
- `status`
- `generated_from_task_id`

Recommended fields:

- `first_physical_action`
- `retry_count`
- `first_seen_on_agenda`
- `last_seen_on_agenda`
- `last_missed_decision_on`
- `completed_at`

## Current Implementation Notes

- Current goal files use frontmatter and may embed `tasks` in goal frontmatter.
- Target behavior keeps Tasks embedded in Goal files and uses Subtasks for breakdown instead of separate task files.
- Current code currently reads `type`, `goalType`, or `description` as a domain-like field.
- V1 Domain grouping should continue to be inferred from Goal frontmatter; Domain metadata files are reserved for richer future behavior.
- Current Goal UI uses priority values such as low, medium, high, and critical. Task and Subtask UI may use `eisenhower_quadrant` for Do, Schedule, Delegate, and Delete classification.
- Current Goal UI should map priority indicators to priority label and color.
- Current subtask breakdown behavior is partially represented in Assistant prompts and persistence; the Assistant panel now surfaces the explicit user decision flow after repeated Subtask misses and persists those choices through vault-backed commands.
- Current Assistant chat can persist user-requested Roadmap Goal additions and metadata edits through assistant-attributed vault mutations.

## Acceptance Criteria

- Roadmap groups active Goals by Domain.
- Roadmap can group active Goals by Domain without requiring Domain metadata files.
- Optional Domain metadata can define icon, color, sort order, urgency/importance thresholds, and archive state when present.
- Goal markdown can represent status, lifecycle, visibility, deadline, success metric, one local owner, a forward-compatible collaborators field, priority, and tasks.
- Tasks remain embedded in Goal markdown files, including when they become complex.
- Users can set an embedded Roadmap Task to recur daily, weekdays, weekly, monthly, or yearly, or clear recurrence back to no repeat.
- Goal priority indicators are color coded and accessible.
- A Task missed for 2 days can be broken into Subtasks.
- A Subtask missed for 2 days triggers the decision flow described above.
- Assistant chat can add a Goal and edit existing Goal metadata in the Roadmap while preserving embedded Tasks/Subtasks.
- Product-facing labels use Goal/Domain, not Objective/goal_type.
